# 项目方案：智能 OJ 题目生成引擎 (AOCE - AI OJ Content Engine)

### 1. 核心愿景

构建一个基于 **LangGraph + Docker Sandbox** 的闭环智能体系统，实现从自然语言描述到“标答代码 + 结构化测试数据”的自动化产出，通过自修复循环（Self-Correction Loop）确保 100% 的代码可执行性。

### 2. 系统架构图

### 3. 核心技术栈

* **编排层**: LangGraph (处理状态流转与重试逻辑)
* **执行层 (沙箱)**: Docker (隔离运行环境) + Python Docker SDK
* **工具层**: MCP (Model Context Protocol) 封装常用算法数据生成工具
* **接口层**: FastAPI + Server-Sent Events (SSE) 实现步骤实时追踪

---

### 4. 详细工作流设计 (LangGraph Nodes)

#### 节点 A: 需求解析 (Parser)

* **输入**: 用户输入的题面文本。
* **任务**: 提取时间/空间限制、输入输出格式（Regex 描述）、变量范围（如 $n \le 10^5$）。
* **状态**: 写入 `State["requirements"]`。

#### 节点 B: 代码双轨生成 (Generator)

* **任务**:
1. 生成最优解标答 `solution.cpp/py`。
2. 生成数据生成器 `generator.py`（需调用预设的 MCP 随机工具）。


* **状态**: 写入 `State["codes"]`。

#### 节点 C: 沙箱验证 (Executor) - **核心环节**

* **动作**:
1. 启动 Docker 容器，挂载临时目录。
2. 运行 `generator.py` 产生 `input.txt`。
3. 运行 `solution.cpp/py` 处理 `input.txt` 产生 `output.txt`。


* **监控**: 记录 CPU 时间、内存峰值、退出码（Exit Code）及 `stderr`。

#### 节点 D: 反思与路由 (Reflector)

* **逻辑**:
* 若执行失败（报错/超时）：分析 `stderr`，将报错信息和失败的代码打包，**回流至节点 B**。
* 若执行成功：抽样几组数据进行逻辑自检。
* **终止条件**: 成功或重试超过 3 次。

### 7. 欠缺补全与深度建议

1. **数据强度校验**:
AI 容易生成“弱数据”。建议在沙箱节点增加一个 **复杂度分析环节**：如果标答在处理 $10^5$ 数据时只用了 1ms，说明 `generator.py` 生成的数据太随机、不够“卡人”。此时应要求 AI 增加“链状图”、“全相等数组”等极端情况。
2. **SPJ (Special Judge) 支持**:
如果是浮点数题目或路径不唯一题目，AI 需要额外写一个 `checker.cpp`。这个 `checker` 同样需要在沙箱里和标答一起交叉验证。
3. **状态持久化**:
利用 LangGraph 的 `checkpointer`。如果 Docker 宿主机压力过大导致任务崩溃，系统可以从上次失败的节点直接恢复，而不是从头解析题面。
